## Known limitations
- **Mock data only**: The app currently uses in-memory mock data (MockDataContext) rather than real database queries. Bookings, desks, rooms, teams, and notifications are all ephemeral — they reset on page refresh. Test assertions about created bookings must happen within the same session.
- **Konva canvas**: The floor map uses react-konva (HTML5 Canvas). Canvas elements are not in the DOM, so standard selectors won't find individual desks. The test runner will need to interact with canvas via coordinates or use the booking wizard's list-based selection.
- **Radix portals**: Dialogs, popovers, and select dropdowns render in Radix portals outside the main component tree. The test runner should query from `document.body`, not within a specific container.
- **Framer Motion animations**: Page transitions and booking wizard step transitions use framer-motion. Tests may need small waits (200-300ms) for exit animations.
- **No real database persistence**: Since all data is mock, "book a desk" doesn't create a real record. Tests verify the UI flow, not backend persistence.
